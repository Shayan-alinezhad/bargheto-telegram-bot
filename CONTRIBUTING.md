# مشارکت در برق تو

از علاقه شما برای مشارکت در پروژه **برق تو** متشکریم. هدف این راهنما، ساده‌کردن گزارش مشکلات، پیشنهاد قابلیت‌ها و ارسال Pull Request است.

## پیش از شروع

- ابتدا `README.md` و `SECURITY.md` را مطالعه کنید.
- Issueهای موجود را جست‌وجو کنید تا موضوع تکراری ایجاد نشود.
- هیچ توکن، رمز عبور، Cookie، فایل `.env` یا اطلاعات کاربران را منتشر نکنید.
- برای آسیب‌پذیری‌های امنیتی از Issue عمومی استفاده نکنید.

## گزارش باگ

از قالب **گزارش باگ** در بخش Issues استفاده کنید و موارد زیر را بنویسید:

1. بخش مشکل‌دار
2. مراحل دقیق بازتولید
3. نتیجه مورد انتظار
4. نتیجه فعلی
5. محیط اجرا
6. Log یا خروجی `/debug` پس از حذف اطلاعات حساس

## پیشنهاد قابلیت

از قالب **پیشنهاد قابلیت جدید** استفاده کنید. علاوه بر راه‌حل پیشنهادی، مسئله‌ای را که قابلیت حل می‌کند توضیح دهید.

## روش مشارکت در کد

1. Repository را Fork کنید.
2. یک Branch جدید بسازید:

```bash
git checkout -b feature/short-description
```

برای رفع باگ می‌توانید از این الگو استفاده کنید:

```bash
git checkout -b fix/short-description
```

3. تغییرات را فقط در فایل‌های مرتبط انجام دهید.
4. کد را از نظر Syntax بررسی کنید:

```bash
node --check worker.js
```

5. تغییرات را Commit کنید:

```bash
git add .
git commit -m "feat: add short description"
```

6. Branch را Push و Pull Request ایجاد کنید.

## استاندارد پیام Commit

ترجیحاً از الگوی Conventional Commits استفاده کنید:

| پیشوند | کاربرد |
|---|---|
| `feat:` | قابلیت جدید |
| `fix:` | رفع باگ |
| `docs:` | تغییر مستندات |
| `style:` | تغییرات ظاهری بدون تغییر منطق |
| `refactor:` | بازآرایی کد |
| `test:` | افزودن یا اصلاح تست |
| `chore:` | نگهداری پروژه |

نمونه:

```text
feat: add outage notification settings
fix: handle source cookie challenge
 docs: improve Cloudflare setup guide
```

## قواعد کدنویسی

- کد باید با محیط Cloudflare Workers سازگار باشد.
- از افزودن وابستگی غیرضروری خودداری کنید.
- نام توابع و متغیرها انگلیسی و خوانا باشند.
- پیام‌های کاربر می‌توانند فارسی باشند.
- ورودی کاربران و داده‌های HTML باید پیش از نمایش امن‌سازی شوند.
- Queryهای D1 باید با Binding پارامتری اجرا شوند.
- مسیرهای مدیریتی باید احراز هویت داشته باشند.
- Secretها فقط از `env` خوانده شوند.

## تغییرات پنل مدیریت

در تغییرات HTML/CSS پنل، موارد زیر را بررسی کنید:

- نمایش صحیح در دسکتاپ و موبایل
- نبودن هم‌پوشانی یا اسکرول افقی صفحه
- خوانایی متن فارسی و چیدمان RTL
- کنتراست مناسب رنگ‌ها
- پشتیبانی از حالت تاریک
- دسترسی‌پذیری دکمه‌ها و فیلدها

## Pull Request

هر Pull Request باید:

- هدف مشخص و محدوده محدود داشته باشد.
- روش آزمایش را توضیح دهد.
- Issue مرتبط را در صورت وجود معرفی کند.
- فایل یا تغییر غیرمرتبط نداشته باشد.
- هیچ اطلاعات حساسی در کد، تصاویر یا Logها نداشته باشد.

## مجوز مشارکت

با ارسال Pull Request، موافقت می‌کنید مشارکت شما تحت مجوز MIT پروژه منتشر شود.
